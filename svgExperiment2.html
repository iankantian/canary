Technically it's a hack. It uses a table as a layout tool. Table is the only mechanism I could find, without massive media query trees, in which you can have elements of fixed and fluid widths mixed together.
The status of volume control is edited by #volumestate.childnode[0] and [1] = % of volume.
The status of progress bar is changed by updating the width percentages of playing and loading.
Resizing is more of a chore than I'd like. The dynamic and varied nature of table renderings between browsers means I couldn't make it just be editing the height of the controls. However, just scaling up the size of elements:

  • play
  • clock
  • mute
  • fullscreen
  • volume position
will do the job, and that can be decided via @media queries, so that's no-code.
We can then remove positionController, all my obtuse fulldivOn, fullscreenOn, fullscreenOff, styleStash, and the DOM modification code with the 'on-resize' listener. This code removal extends to the 'media' type of asset as well. VPAID and FLASH type ads provide their own interfaces, in theory.
